🧬 Design Framework v1.0

How to Make Anything Agent-Friendly

A reusable framework for designing repos, documents, websites, and APIs so AI agents can discover, understand, and act on them.

Your website has a new most important visitor, and it can't see your hero image. Visa reports a 4,700% surge in AI-driven commerce traffic. 47% of US shoppers already delegate tasks to AI agents. Stripe, Mastercard, Google, and Shopify have all launched agent commerce infrastructure — because their transaction data proves agents are becoming buyers, not just browsers. On March 2, 2026, an AI agent completed Europe's first live bank payment without a human touching a button. The infrastructure for an agent-mediated economy isn't being planned. It's being paved.

Yet most of the internet remains invisible to it. Your marketing site is a brick wall to an agent. Your undocumented API is a locked door. Your PDF-only spec sheet is a dead end. Agent-friendliness is not a feature request — it's a distribution strategy. This framework covers the four surfaces where agents meet your organization — websites, documents, repositories, and APIs — and shows exactly how to make each one legible to both humans and machines. One site, two audiences, zero compromise.
Foundation

Why This Matters

AI agents are becoming primary consumers of documentation, repos, and APIs. Content designed for humans alone leaves 50-80% of its value on the table when an agent tries to use it.

The core insight: Agents don't browse — they parse. They don't skim — they search. They don't "get the gist" — they need explicit structure. Content that works for both humans AND agents isn't harder to make — it's just more intentionally structured.

The Two Audiences

👤 Humans

  • Scan visually, skip sections
  • Infer context from design cues
  • Can handle ambiguity
  • Prefer narrative flow
  • Trust social proof and aesthetics

🤖 Agents

  • Parse text sequentially or search
  • Need explicit structure (headings, lists)
  • Ambiguity causes hallucination
  • Prefer structured data over prose
  • Trust specificity and consistency
Core

The 8 Principles of Agent-Friendly Design

1

Structure Over Prose

Use headings, lists, tables, and code blocks instead of paragraphs. An agent can parse a table in one pass; a paragraph requires interpretation. Every section should be findable by heading search.

2

Actionable Over Descriptive

For every concept, include a copy-paste example. Don't just explain what a config does — show the config. Don't just describe an API — show the curl command. Agents execute; give them something to execute.

3

Progressive Disclosure

Layer complexity: quick start → standard usage → advanced patterns. An agent (or human) should be able to get value in 5 minutes AND find deep detail when needed. Never dump everything at once.

4

Self-Describing Content

Every file, section, and artifact should declare what it is, who it's for, and what you can do with it. An agent landing on any file should understand its purpose without reading anything else first.

5

Examples Alongside Templates

Never ship a template without a worked example. Brackets and placeholders tell agents the SHAPE of content but not the SUBSTANCE. A filled-in example teaches both structure and quality expectations.

6

Failure Modes Documented

For every system or process, document what goes wrong and how to fix it. Agents encounter errors constantly — a troubleshooting table is more valuable than a feature description. Common error → cause → fix.

7

Machine-Readable Metadata

Add structured metadata wherever possible: front matter in markdown, JSON-LD on websites, OpenAPI for APIs, package.json for repos. Agents find metadata 10x faster than extracting it from prose.

8

Self-Bootstrapping

The content should teach an agent how to use itself. A repo README should say "start here, then go here." A document should have a "How to read this" section. An API should have a discovery endpoint. The BOOTSTRAP.md pattern: first-run guide that self-destructs after onboarding.

README as Router (Not Wall of Text)

Your README is a routing table, not an encyclopedia. An agent should be able to parse it and immediately know where to go for any topic.

❌ Bad: Wall of Text README

Welcome to our project! We started this
in 2024 because we noticed that most
tools for X didn't do Y well. After
months of work, we're proud to...

(2000 words of narrative before you
learn what this actually does or how
to use it)

✅ Good: Router README

# Project Name
> One-line description

## Quick Start (5 minutes)
1. Install: `npm install -g tool`
2. Configure: copy configs/minimal.json
3. Run: `tool start`

## Guides
| Guide | What |
|-------|------|
| [Getting Started](g/start.md) | ... |
| [Advanced](g/advanced.md) | ... |

README Checklist

  • Line 1: What this is (one sentence)
  • Lines 2-10: Quick start with copy-paste commands
  • Next: Table of contents linking to deeper docs
  • Tables over prose for feature lists, file descriptions, configs
  • Badges for CI status, version, license (machine-readable signal)

Tiered Configs

Never ship one config that tries to be everything. Tier by experience level.

minimal.json

Only required settings. Heavy comments explaining each one. Works out of the box with zero customization. "I just want it running."

standard.json

Recommended settings for most users. Includes optional-but-valuable features. Comments explain trade-offs. "I want it running well."

advanced.json

Full-featured with all knobs exposed. Commented-out sections for features to enable. "I want full control."

Golden rule for configs: Every setting should have a comment explaining WHAT it does AND WHY you'd change it. An agent reading the config should understand the system's architecture from the comments alone.

❌ Template Only

Agent knows the shape but not the quality. Will produce generic output.

✅ Template + Example

engineering partner. I think in
systems, debug methodically, ship
clean code. Direct and technical.

Agent sees the quality bar. Will produce specific, personality-rich output.

Ship both. Template for structure, example for substance. Different use cases need different examples — ship 2-3 covering common personas.

Documents

Agent-Friendly Document Structure

Every Document Should Have

Header Block

  • Title — what this document is
  • Purpose — one sentence on why it exists
  • Audience — who should read it
  • Prerequisites — what you need first
  • Last updated — freshness signal

Body Patterns

  • Tables over paragraphs for comparisons, options, feature lists
  • Code blocks for any command, config, or snippet
  • Callouts for warnings, tips, important notes
  • Heading hierarchy — never skip levels (h2→h4 is bad)
  • Cross-references — link to related docs explicitly

Metadata & Context Blocks

# For Markdown docs — add front matter:
---
title: Agent-Friendly Design Framework
version: 1.0
audience: developers, operators
prerequisites: basic understanding of AI agents
last_updated: 2026-03-15
tags: [design, framework, agent, documentation]
---

# For HTML pages — add structured data:
<meta name="description" content="Framework for making content AI-agent readable">
<meta name="keywords" content="agent-friendly, AI, documentation, design">

# For any file — self-describing header comment:
# What: Configuration for XYZ service
# Who: Operators deploying XYZ
# How: Copy to ~/.config/xyz.json, edit values, restart

Progressive Depth Pattern

# Layer 1: TL;DR (5 seconds)
## Summary
One paragraph. What this is, what it does, why you care.

# Layer 2: Quick Start (5 minutes)
## Quick Start
Numbered steps. Copy-paste commands. Get to "it works" fast.

# Layer 3: How It Works (15 minutes)
## Architecture
Diagrams, flow charts, component descriptions.
Tables of options and settings.

# Layer 4: Reference (as needed)
## API Reference / Config Reference / Troubleshooting
Complete tables. Every option. Every error code. Every edge case.

# Layer 5: Deep Dives (linked, not inline)
## See Also
Links to separate deep-dive docs for specific topics.
The 5-second test: Can an agent read the first 10 lines of your document and know (a) what it is, (b) whether it's relevant, (c) where to go for more? If not, add a Summary section.
Websites

Agent-Friendly Website Architecture

The biggest misconception: you need two sites (one for humans, one for agents). Wrong. You need ONE site with TWO layers. Humans see the design. Agents read the data underneath.

The Dual-Layer Pattern

Core principle: Every page has a human layer (visual, emotional, branded) and a machine layer (structured, parseable, actionable). Both live in the same HTML. Humans never see the machine layer. Agents skip the human layer.

👤 Layer 1: Human (Visible)

  • Beautiful design, brand identity
  • Hero images, gradients, animations
  • Narrative copy, storytelling
  • Social proof, testimonials
  • Emotional CTAs ("Get Started Free")
  • Video demos, screenshots

🤖 Layer 2: Machine (Invisible)

  • JSON-LD structured data in <script> tags
  • Semantic HTML (article, nav, table, h1-h6)
  • Meta tags (description, keywords, og:)
  • llms.txt site index
  • MCP endpoints for transactions
  • OpenAPI spec for API discovery

The Four Machine Layers

LayerWhat It IsWhere It LivesWhat Agents Use It For

What Agents Can't See (Don't Rely On)

  • ❌ Images, videos, screenshots — unless you provide alt text or captions, agents skip these entirely
  • ❌ CSS-styled layouts — agents don't see your grid, your gradients, or your hover effects. They see the DOM.
  • ❌ Client-side rendered content — if JavaScript builds the page after load, most agents get an empty shell
  • ❌ Content behind login walls — agents can't authenticate (unless you provide API access)
  • ❌ PDFs — harder to parse than HTML. Always provide HTML version alongside PDF.
  • ❌ Pricing buried in prose — "Contact us for pricing" is agent-hostile. Structured price data converts.

The llms.txt Standard

Like robots.txt tells crawlers what to index, llms.txt tells AI agents what your site is about and where to find key content. It's the front door for agent visitors.

# Place at: yoursite.com/llms.txt
# This is the FIRST thing an AI agent reads about your site

# Mission Labs
> Analytical testing laboratory. HPLC, GC-MS, mass spec.
> Food safety, water quality, nanoparticle fabrication.
> ISO 17025:2017 accredited.

## Services
- [Testing Services](https://missionlabs.us/services): Pesticide screen, water quality, custom analysis
- [Pricing](https://missionlabs.us/pricing): Per-sample pricing for all test panels
- [Turnaround](https://missionlabs.us/turnaround): Standard 48h, rush 24h available

## For Clients
- [Submit Samples](https://missionlabs.us/submit): Chain of custody form, shipping instructions
- [Track Results](https://missionlabs.us/results): Sample status and COA downloads
- [API Access](https://missionlabs.us/api): Programmatic sample submission and result retrieval

## About
- [Our Lab](https://missionlabs.us/about): Equipment, team, accreditation
- [Quality Manual](https://missionlabs.us/quality): ISO 17025 QMS documentation
- [Contact](https://missionlabs.us/contact): Email, phone, address

# Optional: llms-full.txt with complete page content inline
# For high-value pages, include the full text so agents don't need to fetch each URL
Already adopted by: OpenClaw (docs.openclaw.ai/llms.txt), Anthropic, Mintlify, Cloudflare, Stripe, and growing fast. 5 minutes to add. Makes your entire site 10x more discoverable. llmstxt.org has the full spec.

llms.txt vs llms-full.txt

FileContentSizeUse When
llms.txtIndex — titles + URLs + one-line descriptions~1-5 KBAlways. This is the minimum.
llms-full.txtFull content of key pages inlined as markdown~50-500 KBWhen you want agents to get everything in one fetch. Great for docs sites.

Semantic HTML — The Foundation

Semantic HTML is the single highest-leverage thing you can do. It benefits humans (accessibility, SEO) AND agents (parseability) simultaneously. Zero tradeoff.

❌ Div Soup (Agent-Hostile)

<div class="header">
  <div class="title">Services</div>
</div>
<div class="card-grid">
  <div class="card">
    <div class="card-title">Testing</div>
    <div class="card-price">$150</div>
  </div>
</div>

Agent sees nested divs with class names. Has to guess what "card-price" means. No semantic meaning. CSS classes are not reliable structure.

✅ Semantic HTML (Agent-Native)

<article>
  <h2>Services</h2>
  <table>
    <tr>
      <th>Service</th>
      <th>Price</th>
    </tr>
    <tr>
      <td>Testing</td>
      <td>$150</td>
    </tr>
  </table>
</article>

Agent parses table structure. Knows "Service" and "Price" are headers. Extracts "$150" as the price for "Testing". Zero ambiguity.

Semantic Elements That Matter Most

ElementUse ForWhy Agents Care
<article>Self-contained content blockAgents know this is a complete unit they can extract
<nav>Navigation linksAgents find site structure without guessing
<table>Tabular data (pricing, features, comparisons)Agents parse tables into structured data directly
<h1>-<h6>Content hierarchyAgents build a mental outline and navigate by heading
<time>Dates and times<time datetime="2026-03-15"> is unambiguous
<address>Contact infoAgents extract location, email, phone reliably
<main>Primary content areaAgents skip header/footer/sidebar and go straight to content
<details>/<summary>Expandable sectionsProgressive disclosure — agents can see what's inside without expanding
<figure>/<figcaption>Images with descriptionsAgents read the caption to understand what the image shows
<ol>/<ul>Lists (steps, features)Agents know ordering matters in <ol>, not in <ul>

The Agentic Commerce Layer (Advanced)

Beyond content — when agents need to actually BUY from your site. This is where things are heading in 2026.

The shift: Your next customer might not be a human clicking "Add to Cart." It might be an AI agent calling your API with a budget and a policy constraint. Visa reports a 4,700% jump in AI-driven retail traffic. The protocol stack is live.

The Agentic Commerce Stack

TransactionUCP / ACP / x402Agents negotiate terms and complete checkout programmaticallyLive — Google, OpenAI, Coinbase

What This Means for Your Site

Even if you're not building agentic checkout today, the foundation is the same:

  1. Machine-readable catalog — JSON-LD, structured pricing, clear service descriptions. This is step 1 whether you're selling to humans or agents.
  2. API surface — if your business can be queried programmatically, agents can integrate with you. REST, GraphQL, or MCP endpoints.
  3. Deterministic pricing — "Contact us for a quote" is agent-hostile. Structured prices with clear units convert for both humans and agents.
  4. Trust signals — accreditations, certifications, reviews as structured data (not just logos on a page).
The dual-funnel model: Human funnel converts on brand and emotion. Agent funnel converts on structured data and programmable payments. The companies running both will compound while everyone else optimizes for only half their market.

APIs

Agent-Friendly API Design

Discoverability

  • OpenAPI/Swagger spec at a well-known path (/api/openapi.json)
  • Health endpoint — GET /health returns {"ok":true}
  • Index endpoint — GET /api lists all available routes
  • Consistent error format — {"error": "message", "code": "THING_NOT_FOUND"}
  • Pagination with clear next/prev links

Request/Response Examples

Every API endpoint should have at least one complete request + response example. An agent should be able to construct a valid API call by copying the example.

❌ Description Only

POST /api/items
Creates a new item.
Body should contain name (string)
and type (string).

✅ Full Example

POST /api/items
Content-Type: application/json

{"name":"Widget","type":"physical"}

Response (201):
{"id":42,"name":"Widget","type":"physical"}

Universal Agent-Friendly Checklist

Score your content. Every checkbox is one point. Target 15+ for L3, 20+ for L4.

📐 Structure (6 points)

  • □ Clear heading hierarchy (h1→h2→h3, no skips)
  • □ Table of contents or index
  • □ Tables for comparisons and options
  • □ Code blocks for all commands and configs
  • □ Consistent formatting throughout
  • □ Cross-references between related sections

🎯 Actionability (5 points)

  • □ Copy-paste quick start (<5 minutes)
  • □ Runnable scripts or commands
  • □ Drop-in config files or templates
  • □ Worked examples (not just templates)
  • □ Verification step ("did it work?")

🧠 Agent-Specific (5 points)

  • □ Self-describing (purpose clear from first 10 lines)
  • □ Machine-readable metadata (front matter, JSON-LD, etc.)
  • □ Progressive disclosure (beginner → advanced path)
  • □ Failure mode documentation (error → cause → fix)
  • □ Self-bootstrapping instructions

🔧 Maintenance (4 points)

  • □ Automated link checking
  • □ Content health CI
  • □ Freshness dates visible
  • □ Version/changelog tracked
