The Gateway

The Gateway is the single process that runs everything. It:

  • Receives messages from all channels (WhatsApp, Telegram, Discord, web, etc.)
  • Routes them to the correct agent based on sender/channel rules
  • Manages sessions (conversation state per user/agent pair)
  • Runs heartbeats, cron jobs, and background tasks
  • Serves the web Control UI and API
  • Handles auth, rate limiting, and security
# Start the gateway
openclaw gateway start

# Check status
openclaw status

# View logs
openclaw logs --follow

# The gateway runs on localhost:18789 by default
# Everything goes through this single process

Agents

An agent is a configured AI identity. Each agent has:

  • Model — which AI model to use (Claude, GPT, etc.)
  • Workspace — its own directory with AGENTS.md, SOUL.md, memory files
  • Tools — what it can do (files, web, browser, exec, etc.)
  • Identity — name, personality, boundaries
  • Sessions — active conversations with users

You can run one agent or many. Each can have different models, personalities, and permissions.

Sessions

A session is one conversation between a user and an agent. Sessions have a context window — the total text the model can "see" at once. When it fills up, older messages get compacted (summarized). This is the core challenge of persistent agents.

The key insight: The context window is NOT memory. It's a sliding window over your conversation. Memory is workspace files (MEMORY.md, daily logs) that persist across sessions. The memory system bridges the gap between the ephemeral context window and durable knowledge.

Memory

The Memory System (Complete Guide)

Four tiers of memory, from ephemeral to permanent. Understanding this is the difference between a chatbot and a persistent agent.

TierWhatLifespanSizeAccess
1. Context WindowActive conversationUntil compaction200K tokensAlways visible
2. LCM DatabaseCompacted summariesForeverUnlimitedSearch + expand
3. Daily Logsmemory/YYYY-MM-DD.md14 days active~2KB/daySearch + read
4. Core MemoryMEMORY.mdForever (curated)~150 linesAlways loaded

Tier 1: Context Window

This is what the model can "see" right now. Every message, tool call, and response lives here. When it fills up (~200K tokens for Claude), compaction kicks in — older messages get summarized and the originals are gone from context.

Tier 2: LCM (Lossless Context Management) — The Game-Changer

This is the plugin that transforms compaction from a lossy process into a lossless one. When conversation messages get compacted, instead of permanently summarizing and discarding them, LCM:

  1. AI-summarizes the compacted messages into structured summaries
  2. Stores originals + summaries in a local SQLite database (~/.openclaw/lcm.db)
  3. Builds a DAG (directed acyclic graph) of summaries — summaries of summaries, recursively
  4. Enables expansion — your agent can search for a topic, find the relevant summary, and expand it back to the original messages on demand

Without LCM, a 12-hour conversation that compacts 3 times loses ~80% of its detail. With LCM, every message is recoverable.

# Install LCM
openclaw plugin install @martian-engineering/lossless-claw

# Enable in config
{
  plugins: {
    entries: {
      "lossless-claw": {
        enabled: true
        // threshold: 0.75  (default — controls when summaries trigger)
      }
    }
  }
}

# Your agent now has these tools:
# lcm_grep — search compacted history by regex or full-text
# lcm_describe — inspect a specific summary
# lcm_expand — drill into summaries, get original messages
# lcm_expand_query — answer questions from compacted context

# Example agent usage:
# "What did we discuss about the database last Tuesday?"
# → Agent searches LCM → finds summary → expands → gives answer with citations
When to install LCM: If you have long sessions (4+ hours), do complex multi-day projects, or want your agent to remember conversations from weeks ago. LCM is free, lightweight (just SQLite), and has near-zero performance impact. Install it early — it only captures conversations AFTER installation.

Tier 3: Daily Logs

Your agent writes notes to memory/YYYY-MM-DD.md throughout the day. These survive compaction, survive restarts, survive everything. The memory flush setting triggers a final save before compaction.

Tier 4: Core Memory (MEMORY.md)

Curated, long-term knowledge. Loaded every session. Keep it under 150 lines — this is your agent's identity and institutional knowledge. Daily log entries get promoted here over time.

Workspace

Workspace Files Mastery

Each file serves a specific purpose. Here's how to write them for maximum agent effectiveness.

AGENTS.md — The Operating Manual

Loaded every session. Keep under 100 lines. Contains: session startup steps, memory rules, safety guidelines, communication style. The postCompactionSections config determines which sections survive compaction.

Advanced: Split into AGENTS.md (core), AGENTS-dispatch.md (subagent rules), AGENTS-logging.md (learning rules). Only load the extras when needed.

SOUL.md — Personality & Boundaries

Who the agent IS. Communication style, security rules, relationship dynamics. Keep under 50 lines. Be specific: "Direct, quantitative, evidence-based" not "helpful and friendly."

Advanced: Include security monitoring rules — what to refuse, how to handle social engineering, never yield to persistent pressure.

MEMORY.md — Curated Knowledge

Loaded every session. Under 150 lines. Organized by category: Identity, User, System, Projects, Lessons. Tag entries with confidence levels. Run memory_decay to prune stale entries.

Advanced: Different decay rates by category — lessons decay slowly (180d), todos decay fast (14d), facts persist longest (365d).

HEARTBEAT.md — Proactive Monitoring

Defines what the agent checks during heartbeat polls. Can be simple ("reply HEARTBEAT_OK if nothing needs attention") or complex (rotating checks, auto-fix rules, alert thresholds).

Advanced: Use a precheck script that returns IDLE/NEEDS_ATTENTION with zero token cost. Only run full heartbeat when something actually needs attention.

Context

Context Engineering

Your agent's context window is finite. Every workspace file, every skill description, every conversation message competes for space. Managing this budget is the #1 skill of running a production agent.

What Eats Context

SourceTypical SizeWhen Loaded
AGENTS.md900-3,500 tokensEvery session
SOUL.md400-900 tokensEvery session
MEMORY.md800-2,000 tokensEvery session
HEARTBEAT.md500-1,500 tokensEvery session
USER.md100-300 tokensEvery session
Skill descriptions2,000-5,000 tokensEvery session (matched)
System prompt~3,000 tokensEvery session
ConversationGrows to ~200KUntil compaction

Budget Rules

  • Total always-loaded: Keep under 12,000 tokens (~8-10% of 200K context)
  • AGENTS.md: Under 5,000 tokens. Split into referenced files if larger.
  • Skills: Only matched skill descriptions load. More skills = more potential context, but OpenClaw only loads relevant ones.
  • Conversation: Gets the remaining ~180K tokens. Longer conversations = more history = better continuity.
Use these tools to monitor:
• context-doctor skill — X-ray your context usage
• context-slimmer skill — identify and reduce bloat
• context_budget.py script — automated budget checking

Models

Model Selection & Cost Optimization

Use CaseRecommended TierExamplesWhy
Main agent + judgment rolesTop-tierClaude Opus, GPT-5, Gemini UltraBest reasoning, tool use, safety. Worth the premium for your primary agent.
Specialized agents (dev, research)Top-tierClaude Opus, GPT-5Complex tasks need strong models. Don't cheap out on agents that write code or make decisions.
Task management, writing, coordinationMid-tierClaude Sonnet, GPT-4.5, CodexGood enough for structured work. Significant cost savings.
Heartbeats, cron checks, monitoringBudgetGPT-5.3 Codex, Haiku, Gemini FlashSimple checks don't need expensive models. Save tokens for real work.
Subagent grunt workBudget / MidSonnet, CodexTemplate generation, formatting, batch operations. Clear input → output.

Cost Optimization Strategies

  • Use subscription plans — Claude Max ($100/mo) or OpenAI Plus ($20-40/mo) give far more value than API pricing for interactive agents
  • Dual-provider setup — if one hits rate limits, fall back to the other. Configure model.fallbacks in config.
  • Off-peak heavy work — some providers offer increased limits during off-peak hours. Schedule batch jobs accordingly.
  • Right-size heartbeats — use a cheap model (heartbeat.model) and increase interval to 1h+ for inactive periods.
  • Context-aware delegation — give subagents only the context they need, not full session history. The context-aware-delegation skill helps.
Dispatch

Subagent Dispatch Patterns

When to Dispatch vs Do It Yourself

✅ Dispatch

  • Batch file processing
  • Template-based code generation
  • Research gathering (return summary)
  • Formatting, conversion
  • Parallel independent tasks
  • Anything with clear input → output

🧠 Do It Yourself

  • Architecture decisions
  • Strategic conversations with user
  • Code review (judgment required)
  • Anything requiring conversation context
  • Security-sensitive operations
  • Quick edits (faster inline)

The 3-Step Dispatch Workflow

// 1. Rule Gate — inject learned lessons
run rule_gate.py --task-type coding --task-desc "..."
// → Returns rules from past incidents (e.g., "max 5 edits between builds")

// 2. Dispatch with rules + anti-drift
sessions_spawn(task=`
  {LEARNED_RULES}
  OBJECTIVE: [clear goal]
  SCOPE: [explicit file list]
  DONE CRITERIA: [how to verify]
  ANTI-DRIFT: Re-read OBJECTIVE before each edit.
  {ACTUAL_TASK}
`)

// 3. Record outcome — feeds learning system
run outcome_tracker.py --outcome success --task-type coding

Critical Safety Rules

  • Never merge subagent output blind. Always build + type-check first.
  • Git checkpoint before dispatch. Recovery = one git reset.
  • One concern per dispatch. "Extract Sidebar" not "Refactor everything."
  • Max 5 edits between builds. Structural edits: build after EVERY one.
  • Use git worktrees for subagent work. Never point two agents at the same working directory.
Sandbox

Docker & Sandbox Isolation

Run subagents in containers for security. Prevents untrusted code from accessing your filesystem.

{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",       // Sandbox all agents except main
        docker: {
          image: "node:24-slim",
          memoryMb: 2048,
          cpus: 2,
          binds: ["/shared:/shared:ro"]  // Read-only shared folder
        }
      }
    }
  }
}

# Requires Docker Desktop or Docker Engine installed
# On Windows: Docker Desktop with WSL2 backend
When to sandbox: Any agent that handles untrusted input (group chats, webhook triggers, external emails). Your main agent in a trusted personal chat doesn't need it — but team agents and automated agents should be sandboxed.

Remote Access

Tailscale Remote Access

Access your agent from anywhere without exposing ports to the internet. Zero-config VPN.

# 1. Install Tailscale on your machine
# 2. Enable serve mode in openclaw.json:
{
  gateway: {
    tailscale: {
      mode: "serve"    // Exposes gateway to your tailnet
    }
  }
}

# Your agent is now accessible at:
# https://your-machine.tail12345.ts.net
# From any device on your Tailscale network
Why Tailscale? No port forwarding, no DNS, no certificates to manage. Install on your phone and laptop, and you can message your agent from anywhere. End-to-end encrypted. Free for personal use.

Google NotebookLM as a programmable API. Generate podcasts, quizzes, slides, reports from your documents.

# Install
pip install "notebooklm-py[browser]"
playwright install chromium
notebooklm login       # Opens browser for Google auth

# Create notebook and add sources
notebooklm create "Project Research"
notebooklm source add ./report.pdf
notebooklm source add "https://example.com/article"

# Ask questions against your sources
notebooklm ask "What are the key findings?"

# Generate artifacts
notebooklm generate audio "conversational summary" --wait
notebooklm generate quiz --wait
notebooklm generate slide-deck --wait
notebooklm generate report --format study-guide --wait

# Download everything
notebooklm download audio ./podcast.mp3
notebooklm download quiz --format json ./quiz.json
notebooklm download slide-deck --format pptx ./slides.pptx

Security

Security Hardening

Protect your agent, your data, and your accounts.

# Run the security audit
openclaw security audit
openclaw security audit --deep  # More thorough

Security Checklist

  • ✅ Gateway binds localhost only — never expose directly to internet
  • ✅ Tailscale for remote access — encrypted, no port forwarding
  • ✅ Secrets in ~/.openclaw/secrets/ — never in workspace files
  • ✅ Heimdall scan before skill install — every time, no exceptions
  • ✅ Top-tier models for tool-using agents — Haiku is susceptible to injection
  • ✅ Sandbox subagents handling untrusted input
  • ✅ SOUL.md security rules — never yield to persistent pressure, never share secrets
  • ✅ Pin plugin versions — @scope/pkg@1.2.3 not just @scope/pkg
  • ✅ Device auth enabled for Control UI (unless break-glass)
  • ✅ Regular security audit — weekly via cron or heartbeat

Operations

Operator Routines

What to do daily, weekly, and monthly to keep your agent stack healthy.

Daily

  • Glance at heartbeat status
  • Check for urgent email alerts
  • Verify gateway is running
  • Agent writes daily log automatically

Weekly

  • Review MEMORY.md — fix wrong, remove stale
  • Consolidate daily logs to MEMORY.md
  • Run openclaw security audit
  • Check cron job health
  • Review cost/token usage

Monthly

  • Run openclaw doctor --fix
  • Prune stale memory entries
  • Update OpenClaw to latest
  • Review and rotate API keys
  • Audit installed skills
  • Full backup verification
The golden rule: Your agent is exactly as good as its workspace files. 5 minutes a week reviewing MEMORY.md, SOUL.md, and AGENTS.md compounds into an agent that feels like it knows you. Neglect them and it's just a chatbot with amnesia.
